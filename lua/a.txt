● Сега имам пълната картина. Намерих два реални проблема:

  ---
  Проблем 1 — Снимката config.highlights може да има nil цветове

  config/highlights.lua вика build() при зареждане на модула (преди
  highlight.setup()). Ако по това време lvim-utils.colors не е синхронизирал
  палитрата, всички bg/fg са nil → {bg=nil, fg=nil} в Lua е {} → nvim_set_hl(0,
   name, {}) → празна група.

  group_exists проверява с vim.tbl_isempty(hl) — празна група се приема за
  несъществуваща. При следваща смяна на colorscheme apply_all опитва пак → пак
  {} → groupите остават празни докато colors.on_change не светне.

  Поправка: в M.setup() да се вика config.build_highlights() вместо статичния
  config.highlights.

  ---
  Проблем 2 — on_change прави регистрацията постоянно force=true

  При lhl.register(config.build_highlights(), true) регистърът се обновява с
  force=true за всяка група. От този момент apply_all при всяка смяна на
  colorscheme винаги прилага с force — вече не се спазва "само ако не е
  дефинирана".

        LvimSpaceSign = { bg = c.bg_dark, fg = c.blue },
        LvimSpaceFuzzyPrimary = { bg = c.bg_dark, fg = c.red, bold = true },
        LvimSpaceFuzzySecondary = { bg = c.bg_dark, fg = c.magenta, bold =

    }
end
